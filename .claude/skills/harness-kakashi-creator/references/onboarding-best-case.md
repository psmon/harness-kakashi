# 온보딩 베스트 케이스 — "피라미드를 만들었을 뿐인데"

> 이 문서는 기본 하네스에 설치되지 않는다.
> 온보딩(Case 2-A) 시 정원지기가 **참고 사례**로 인용할 수 있는 레퍼런스다.

---

## 이 사례의 주인공

개발 경험이 많지 않은 사용자가 카카시 하네스를 처음 사용했다.
요청은 단순했다:

> "hello world 란 단어를 가지고 피라미드 도형을 연출해줘"

그 뒤에 일어난 일을 시간순으로 따라가 보자.

---

## Scene 1: 씨앗을 심다 — 코드 생성

```
사용자: 카카시 하네스~ projects 하위에 닷넷 콘솔프로젝트 생성해줘.
        hello world 피라미드 도형을 연출해줘. 자율연출해.
```

카카시 하네스가 한 일:
- .NET 콘솔 프로젝트 `projects/HelloPyramid/` 생성
- "HELLO WORLD"로 피라미드 + 다이아몬드 + 역피라미드 + 배너 4단계 아스키 아트 구현
- 레인보우 7색 콘솔 컬러 적용
- `dotnet run` 으로 실행 확인까지 완료

**사용자가 한 일: 한 줄 요청.**
**카카시가 한 일: 173줄 코드 작성 + 빌드 + 실행 확인.**

```
실제 로그: harness/logs/execution/2026-04-19-hello-pyramid-dotnet-console.md
```

---

## Scene 2: 꽃을 피우다 — 전체 평가

```
사용자: 카카시 하네스~ 이 프로젝트 전체평가 수행해줘.
```

카카시 하네스가 한 일:
5명의 전문가 에이전트가 **동시에** 코드를 분석했다.

| 전문가 | 한 일 | 결과 |
|--------|-------|------|
| 성능 정찰병 | 2-Pass 성능 안티패턴 스캔 | 이슈 없음 (Info 3건) |
| 테스트 감시병 | 테스트 부재 탐지, 갭 분석 | **테스트 없음 — D등급** |
| 보안 경비병 | OWASP Top 10 전 항목 점검 | 안전 (Low 1건) |
| 빌드 의사 | 빌드 경고/에러 + 안티패턴 스캔 | 깨끗 (0건) |
| 코드 현대화 전문가 | 최신 C# 패턴 적용 여부 | 75% 현대화 완료 |

**종합 등급: B+**

```
실제 로그: harness/logs/full-review/2026-04-19-hello-pyramid-full-evaluation.md
```

## Scene 3: 코칭을 받다 — 구체적 개선 안내

전문가들이 준 피드백은 추상적인 조언이 아니었다.
**코드의 몇 번째 줄을 어떻게 고치라**는 구체적 지시였다:

### 코드 현대화 전문가의 코칭

> "173줄이 전부 Main에 있습니다.
> 4개 Phase를 각각 메서드로 분리하세요:"

```csharp
// 현대화 전문가가 제안한 구조
static void DrawPyramid(string phrase, int height, ConsoleColor[] colors) { ... }
static void DrawDiamond(string phrase, int height, ConsoleColor[] colors) { ... }
static void DrawReflection(string phrase, int height, ConsoleColor[] colors) { ... }
static void DrawBanner(string banner, int totalWidth, ConsoleColor[] colors) { ... }
```

### 테스트 감시병의 코칭

> "출력 로직을 StringWriter로 캡처 가능한 메서드로 분리하면
> 각 도형별 출력을 자동으로 검증할 수 있습니다."

### 보안 경비병의 코칭

> "args를 선언만 하고 사용하지 않아 인젝션 경로가 차단되어 있습니다.
> 만약 향후 사용자 입력을 받게 된다면 — 그때 다시 점검하겠습니다."

**사용자가 배운 것:**
- 왜 메서드를 분리해야 하는지 (테스트 가능성)
- 현대적인 C# 코드 구조가 어떤 것인지
- 보안 관점에서 "안전한 코드"란 무엇인지

## Scene 4: 보안 리포트 — 문서화

```
사용자: 보안평가를 별도 문서로 만들어줘. README에 연결해줘.
```

보안 경비병이 공식 리포트를 작성했다:
- OWASP Top 10 전 항목 점검표
- 공격 표면 분석 (API, 입력, 파일, 네트워크 매핑)
- 심각도 분류 (Critical/High/Medium/Low)
- 3축 정량 평가


## 무엇이 특별한가

이 전체 과정에서 사용자는 **코드를 한 줄도 직접 쓰지 않았다.**

```
사용자 입력            카카시 하네스의 응답
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
"피라미드 만들어줘"  → 173줄 코드 생성 + 빌드 + 실행
"전체평가 해줘"      → 5명 전문가 동시 투입, 8건 발견, 등급 산출
"보안 문서 만들어줘" → OWASP 기반 공식 리포트 + README 연결
```

그리고 이 과정에서 사용자는 자연스럽게 배웠다:

| 카카시가 가르쳐준 것 | 교과서에서의 이름 |
|---------------------|------------------|
| "메서드를 분리하세요" | 단일 책임 원칙 (SRP) |
| "StringWriter로 캡처하세요" | 테스트 가능한 설계 (Testable Design) |
| "args를 사용하지 않아 안전합니다" | 공격 표면 최소화 (Attack Surface Reduction) |
| "컬렉션 표현식을 잘 쓰셨네요" | 최신 언어 기능 활용 (Language Modernization) |
| "preview SDK는 프로덕션에 부적합" | 의존성 관리 (Dependency Management) |

**코드를 만들어달라고 했을 뿐인데, 시니어 개발자 5명에게 코드 리뷰를 받은 셈이다.**

---

## 이 사례를 재현하려면

```
# 1. 정원을 연다
/harness-kakashi-creator init

# 2. 전문가를 심는다 (온보딩에서 제안하는 대로 수락)
/harness-kakashi-creator 새 에이전트 추가해

# 3. 코드를 만든다 (무엇이든)
/harness-kakashi-creator 프로젝트 만들어줘

# 4. 코칭을 받는다
/harness-kakashi-creator 전체 점검해
```

네 줄이면 된다.
정원을 열고, 꽃을 심고, 코드를 만들고, 코칭을 받는다.

## 참고: 하네스 없이도 코드는 만들 수 있다

Claude에게 직접 "피라미드 만들어줘"라고 해도 코드는 나온다.
차이는 **그 다음**이다.

| | 하네스 없이 | 카카시 하네스 |
|---|-----------|-------------|
| 코드 생성 | O | O |
| 전문가 리뷰 | X | 5명 동시 |
| 보안 점검 | X | OWASP Top 10 |
| 성능 분석 | X | 2-Pass 스캔 |
| 개선 코칭 | X | 구체적 코드 제안 |
| 문서화 | X | 공식 리포트 |
| 로그 | X | 모든 활동 자동 기록 |

하네스는 코드를 만드는 도구가 아니다.
**코드를 더 잘 만들도록 돕는 정원**이다.
