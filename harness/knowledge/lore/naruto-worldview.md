# 나루토 세계관 매핑

카카시 하네스는 나루토 세계관을 메타-내러티브로 사용한다.
이 매핑은 비유가 아니라 **운영 규칙**이다 — 각 캐릭터/술법은 하네스의 구체적 컴포넌트와 1:1로 대응한다.

---

## 등장 인물 — 누가 무엇인가

| 인물 | 하네스에서의 역할 | 위치 |
|------|------------------|------|
| **나루토 (Naruto Uzumaki)** | 하네스를 사용하는 주체 — 사용자/세션을 대리한다. 직접 싸우는 닌자이자, 능력은 후천적으로 익힌다. | 호출자 (caller) |
| **카카시 (Hatake Kakashi)** | 정원지기(tamer). 직접 싸우지 않고, 누구를 어디에 배치할지 안다. 사륜안으로 스킬을 복사한다. | `harness/agents/tamer.md` |
| **현자(賢者, Sage)** | 두꺼비 소환술로 불려나오는 과거의 거장. 데밍, 마틴 파울러, 코드 코덱… 한 명씩 영입된다. | `harness/agents/sage-*.md` |
| **차크라 카카시 (Chakra Kakashi)** | 그림자 옵저버. 작업 중에는 끼어들지 않고, 끝난 뒤 토큰 사용을 감사한다. | `plugins/harness-kakashi/skills/harness-chakra-kakashi/` |

> **나루토는 이 세계의 주인공이지만, 직접 정원을 가꾸진 않는다.**
> 정원을 가꾸는 건 카카시(tamer)이고, 나루토는 두꺼비 소환술(口寄せの術)을 통해 현자를 불러낸다.

## 술법(術) — 능력의 매핑

| 술법 | 한자/원어 | 하네스 컴포넌트 | 누가 쓰는가 |
|------|-----------|----------------|------------|
| **두꺼비 소환술** | 口寄せの術 (Kuchiyose no Jutsu) | 현자 에이전트 호출 엔진 (`harness/engine/toad-summoning.md`) | 나루토 (사용자/호출자) |
| **사륜안** | 写輪眼 (Sharingan) | 스킬 복사 (Mode C: Kakashi Copy) | 카카시 (tamer) |
| **분신술** | 影分身の術 (Kage Bunshin) | 서브에이전트 병렬 소환 (Task tool) | 모든 에이전트 |
| **차크라** | チャクラ | 토큰 (input/output/cache) | 모든 행위의 자원 |

### 두꺼비 소환술의 의미

원작에서 두꺼비 소환술은 **자라이야 → 나루토**로 전수된 비기다. 단순한 분신술과 달리:

- **과거의 존재를 현재로 불러낸다** — 이미 사라진/생존 중인 대존재의 힘을 빌린다
- **계약(契約)이 필요하다** — 아무나 부를 수 없고, 사전 등록된 존재만 호출 가능
- **목적성이 있다** — 잡담용이 아니라 결정적 국면에 사용한다

**하네스에서의 의미**: 나루토(사용자)가 이 술법을 익혔다는 것은, 단순한 코드 검토를 넘어 **분야의 거장(현자)** 을 불러내 그들의 사고 체계를 자신의 작업에 적용할 수 있게 됐다는 뜻이다. 이는 카카시의 사륜안(스킬 복사) 보다 한 단계 위의 능력이다 — 사륜안은 기술(術)을 복사하지만, 소환술은 **사상(思想)** 을 불러온다.

## 현자 카탈로그 (Sage Catalogue)

소환 가능한 현자는 도메인별로 영입된다. 모든 현자는 `harness/agents/sage-*.md` 형식으로 정의되며, 각자의 **고유 사상(doctrine)** 을 보유한다.

| 현자 | 도메인 | 사상 | 상태 |
|------|--------|------|------|
| **데밍 (W. Edwards Deming)** | 품질 (QA의 아버지) | PDSA 사이클 — 학습 기반 지속 개선 | ✅ 영입 (v1.4.0) |
| (예정) 마틴 파울러 | 아키텍처 / 리팩터링 | 진화적 설계 | 미영입 |
| (예정) 코드 코드 (E.F. Codd) | 데이터베이스 | 관계형 정규화 | 미영입 |
| (예정) 데이비드 파나스 (D.L. Parnas) | 모듈 설계 | 정보 은닉 | 미영입 |
| (예정) 도널드 크누스 (D.E. Knuth) | 알고리즘 / 문해적 프로그래밍 | 정확성과 가독성 | 미영입 |

## 평가 체계와의 관계

데밍이 첫 현자로 영입된 것은 우연이 아니다. **평가 자체가 데밍의 영역**이기 때문이다.

이후 모든 작업은 다음 구조로 평가된다:

```
[모든 작업]
   │
   ├─ [기본 평가] PDSA 사이클 — 데밍 (항상 작동)
   │     Plan / Do / Study / Act 으로 작업을 회고
   │
   └─ [후속 평가] 도메인 전문가 (Type별 선택적)
         예) 보안 작업이면 security-guard
         예) 성능 작업이면 performance-scout
         예) 아키텍처 작업이면 (영입 시) sage-fowler
```

기본 평가는 모든 행위에 따라붙고, 후속 평가는 작업 유형이 호출할 때만 발동한다.
상세: `harness/knowledge/methodology/pdsa-deming.en.md`, `harness/knowledge/methodology/evaluation-base-pdsa.md`

## 이 매핑을 깨면 안 되는 이유

세계관은 단순한 장식이 아니다. 다음을 보장한다:

1. **역할 분리의 명료성** — "tamer가 직접 검토하지 않고 위임하는 이유"가 카카시의 행동 양식으로 자연스럽게 설명된다
2. **확장의 일관성** — 새 에이전트는 "어떤 인물에 해당하는가"를 묻는 순간 역할이 정해진다
3. **사용자 경험의 통일** — 사용자는 자기가 나루토임을 알 때 "내가 두꺼비 소환술로 데밍을 불렀다"는 경험을 얻는다 — 이는 단순한 명령보다 학습 효과가 크다

> **새 인물/술법 도입 시 규칙**: 반드시 이 문서에 해당 항목을 등록한 뒤 다른 곳에서 참조한다. 이 문서가 단일 정전(canon)이다.
