# 🥷 술법 패턴 (Jutsu Patterns)

> 효율 술법(DO) vs 차크라 낭비 안티패턴(DON'T) — 차크라 카카시의 평가 근거 지식.

---

## 1. 효율 술법 (DO)

### 1.1 차크라 순환 — 캐시 보존 (Circulation)

동일 컨텍스트를 유지하는 한 turn 동안, 시스템 프롬프트/CLAUDE.md/도구 정의는 캐시에 재사용된다.
중간에 **파일 쓰기**, **환경 변경**, **대규모 읽기**가 발생해도 기본 구조는 유지되도록 한다.

**DO**:
- 같은 파일을 다시 볼 때는 이미 읽은 내용을 회상하고 재독하지 않는다.
- 여러 파일을 탐색할 때는 **한 번의 Read 호출에서 offset으로 필요 구간만** 가져온다.
- 병렬로 독립적인 Read/Grep을 한 메시지에 묶는다.

**기준**:
- 캐시 히트율 (cache_read / (cache_read + cache_creation)) ≥ 70% 유지.

### 1.2 술법 정밀도 — 도구 호출 절제 (Tool Economy)

**DO**:
- Grep을 하나 잘 던져서 5곳을 찾는다 (shotgun grep 금지).
- 파일 구조가 명확하면 Glob → 대상 3~5개만 Read.
- 쉘 작업은 하나의 체이닝으로: `cmd1 && cmd2 && cmd3`.
- 순서가 없는 탐색은 한 메시지 안에 병렬 호출.

### 1.3 인술 선택 — 모델 단계화 (Model Tiering)

| 작업 유형 | 권장 모델 | 이유 |
|----------|----------|------|
| 단순 파일 조회, 패턴 매칭 | Haiku | 차크라 소모 최소 |
| 일반 편집·리팩터링 | Sonnet | 품질·비용 균형 |
| 아키텍처 설계, 복잡 디버깅 | Opus | 깊은 추론 필요 시 |

**DO**: 서브에이전트 소환 시 작업 성격에 맞는 모델 지정 (`model: haiku` 등).

### 1.4 분신술 절제 — 서브에이전트 소환 (Subagent Restraint)

**서브에이전트 = 분신술**: 별도 컨텍스트로 실행되므로 차크라를 분리 소모한다.

**DO 조건** (아래 중 하나 이상):
- 광범위 코드 탐색으로 3턴 이상 grep/read가 필요할 때 → `Explore` 또는 `general-purpose`
- 부모 컨텍스트에 남기고 싶지 않은 노이즈 생성 작업
- 독립적이고 병렬 가능한 여러 작업

**DON'T**:
- 단일 파일 읽고 답할 수 있는 질문에 서브에이전트 소환
- 부모가 이미 아는 컨텍스트를 재탐색시키는 소환

### 1.5 술법 집중도 — 프롬프트 경제성 (Prompt Economy)

**DO**:
- CLAUDE.md에 안정적 규약만 넣는다. 실험 내용, 회의 노트 금지.
- 임시 메모는 하네스 노트(harness-note-kakashi)로 분리한다.
- 컨텍스트 압축 전에 중요 결정을 메모리에 저장한다.

## 2. 차크라 낭비 안티패턴 (DON'T)

### 2.1 차크라 고갈 — Cache-Read Storm

**증상**: `cache_read_input_tokens`가 매 턴 급증 (수십 M 토큰), 실제 작업은 작음.
**원인**: CLAUDE.md나 시스템 프롬프트에 거대 문서가 반복 주입되는 경우.
**진단**: `/context`로 CLAUDE.md 비중 확인. 30% 이상이면 위험.
**처방**: CLAUDE.md 슬림화, 세부는 `Docs/` 하위로 분리.

### 2.2 순환 파괴 — Cache Busting

**증상**: `cache_creation_input_tokens`가 매 턴 생성되고 `cache_read`는 작음.
**원인**: 매 턴마다 도구 정의가 바뀜 (플러그인 토글, skill 로드·언로드), 또는 시스템 메시지 변경.
**처방**: 세션 중 plugin·skill·MCP 변경 최소화.

### 2.3 반복 독서 — Re-Read Loop

**증상**: 같은 파일을 3회 이상 Read. 또는 연속 Grep으로 점점 좁혀가지 못하고 뻗음.
**원인**: 이전 결과를 기억하지 않고 재확인.
**처방**: 핵심 부분은 대화 맥락에 요약으로 붙여두고 재호출 금지.

### 2.4 분신술 남발 — Agent Spam

**증상**: 한 세션에서 5개 이상 서브에이전트 소환, 각 에이전트가 사소한 작업.
**원인**: "병렬이면 빠를 것"이라는 오해 + 컨텍스트 분리 욕구.
**처방**: 부모가 1~2번 Grep으로 끝낼 일은 본인이 처리.

### 2.5 인술 불일치 — Model Mismatch

**증상**: Haiku로 충분한 정형 작업(로그 필터, CSV 파싱)을 Opus로 수행.
**원인**: 기본 모델이 Opus로 설정된 상태 방치.
**처방**: 서브에이전트 호출 시 `model: haiku` 명시.

### 2.6 술법 중복 — Duplicate Jutsu

**증상**: `Bash(git status)` → `Bash(git diff)` → `Bash(git log)` 순차 호출 (1개 메시지에 병렬 가능).
**원인**: 병렬 호출 습관 부재.
**처방**: 독립 호출은 단일 메시지에 묶어 병렬.

---

## 3. 평가 힌트

차크라 카카시가 위 패턴을 실제 세션에서 탐지하는 힌트:

| 안티패턴 | 탐지 힌트 |
|---------|----------|
| Cache-Read Storm | JSONL `usage.cache_read_input_tokens` 평균 > 100k/턴 |
| Cache Busting | `cache_creation` 이 매 턴마다 재생성 |
| Re-Read Loop | 동일 파일 경로를 가진 Read 도구 호출 ≥ 3회 |
| Agent Spam | Agent 도구 호출 ≥ 4회, 각 작업 범위가 좁음 |
| Model Mismatch | 서브에이전트 로그에서 `Opus` + 단순 작업 |
| Duplicate Jutsu | 순차 메시지에서 Bash 3회 이상 개별 호출 |

---

## 4. 요약

> 많이 쓴다고 강해지는 게 아니라 **잘 써야 이긴다**.
> 캐시는 차크라 순환. 끊기면 차크라를 새로 소모한다.
> 분신술은 강력하지만 차크라를 N배로 쓴다. 꼭 필요할 때만.
> 상급 인술은 기본값이 아니다. 작업 난이도에 맞춰 단계를 선택하라.
