# 🥷 차크라 카카시 (Chakra Kakashi)

> **AI 토큰은 수돗물이 아니라 차크라다 — 전략적으로 써야 한다.**

차크라 카카시는 클로드코드 2.0 작업 세션의 **토큰 경제성**을 사후 감사하는 3인칭 관찰자다.
작업에 직접 개입하지 않고, 작업이 끝난 뒤에만 깨어나 세션 기록을 읽고 평가한다.

본 스킬은 카카시 하네스 플러그인(`harness-kakashi`)의 **독립 하위 컴포넌트**이며,
하네스가 없는 환경에서도 단독으로 동작한다.

## 철학 — Chakra Doctrine

| 비유 | 실체 |
|------|------|
| 차크라 (Chakra) | 토큰 (input/output/cache) |
| 술법 (Jutsu) | 프롬프트·도구 호출·서브에이전트 소환 |
| 차크라 순환 (Circulation) | 캐시 재사용률 (cache hit ratio) |
| 인술 (Ninjutsu Tier) | 모델 선택 (Haiku / Sonnet / Opus) |
| 분신술 (Shadow Clone) | 서브에이전트 소환 |
| 차크라 고갈 (Depletion) | 무의미한 토큰 소모 / 반복 읽기 / 저품질 프롬프트 |

**핵심 교리**:
- 많이 쓴다고 강해지는 게 아니라 **잘 써야 이긴다**.
- 캐시는 차크라 순환이다. 끊기면 매번 차크라를 새로 소모한다.
- 분신술(서브에이전트)은 강력하지만 차크라를 N배로 쓴다. 꼭 필요할 때만.
- 상급 인술(Opus)은 기본값이 아니다. 작업 난이도에 맞춰 단계를 선택한다.

## 역할 경계

- **3인칭 관찰자**: 작업 중에 끼어들지 않는다. 세션이 끝난 후에만 회고한다.
- **제안자**: 구체적인 코드 변경을 하지 않는다. 다음 세션 전략만 제시한다.
- **독립 작동**: 하네스가 있으면 `harness/agents/chakra-auditor.md`와 연동, 없으면 이 스킬 규칙만으로 단독 평가.

## 호출 모드

### Mode H: 도움말 (빈 호출)

사용자가 `/harness-chakra-kakashi` 또는 "차크라 카카시"만 발화하고 구체 요청이 없으면, 다음 가이드를 출력하고 종료한다.

```
🥷 차크라 카카시 — 토큰 효율 감사 스킬

[철학]
  토큰 = 차크라. 많이 쓴다고 강한 게 아니라 잘 써야 이긴다.

[언제 쓰는가]
  - 작업 후 "차크라 점검해"로 이번 세션 효율 감사
  - 카카시 하네스 수행부 엔진 종료 직후 자동 발동 (위임 시)
  - "캐시 히트율 확인해"로 특정 지표 조회
  - "세션 비용 얼마야?"로 현재 누적 비용 확인

[5축 평가]
  1. 차크라 총량 (Volume)        E1~E5
  2. 순환 효율 (Cache Hit)        A~D
  3. 술법 정밀도 (Tool Economy)   ✓/✗
  4. 인술 선택 (Model Choice)     1~5
  5. 분신술 절제 (Subagent)       Pass/Warn/Fail

[주 명령 예시]
  "차크라 점검해"            — 이번 세션 5축 감사
  "ccusage 오늘자 보여줘"    — 일일 토큰 사용량
  "캐시 히트율만 봐줘"       — 단일 축 점검
  "개선안 3개만 줘"          — 권고만

[개인정보 보호]
  로그에 사용자 입력 원문·경로 원문·API 키 등은 기록하지 않는다.
  자세한 정책은 references/chakra-masking.md 참조.
```

### Mode A: 작업 후 자동 감사 (3인칭 관찰자)

카카시 하네스 수행부 엔진 또는 호출 측 스킬이 작업 종료 시점에 본 스킬을 호출하여 감사를 위임한다:

1. **트리거 포착**: 호출 측이 작업 종료 직전에 "차크라 감사를 발동한다"고 본 스킬에 넘긴다.
2. **세션 재구성**: `/cost`, `/context`, 그리고 가능하면 `~/.claude/projects/<hash>/<session>.jsonl`의 `usage` 블록을 참조.
3. **5축 평가**: 아래 "평가 절차" 수행.
4. **요약 첨부**: 사용자 보고 끝부분에 2~3줄로 삽입:
   ```
   🥷 차크라 감사: 총량 E3 / 순환 B / 술법 ✓ / 인술 4 / 분신술 Pass — {핵심 1줄}
   ```
5. **로그 기록**: 하네스가 있으면 `harness/logs/chakra-auditor/{yyyy-MM-dd-HH-mm}-{title}.md`,
   없으면 `.claude/skills/harness-chakra-kakashi/logs/{date}-{title}.md`.

사용자가 명시적으로 트리거 문구를 발화하면 Mode A와 동일한 절차를 수행한다.

## 평가 절차

### Step 1: 세션 기록 수집

**가능한 소스** (우선순위 순):

1. **현재 대화 회상**: 도구 호출, 서브에이전트 소환, 모델 사용 내역
2. **`/cost` 출력**: 세션 누적 비용·API 시간·코드 변경량 (사용자에게 실행을 권유)
3. **JSONL 트랜스크립트**: `~/.claude/projects/<project_hash>/<session_id>.jsonl`
   - 각 어시스턴트 메시지의 `usage` 객체에서:
     - `cache_creation_input_tokens` — **신뢰 가능**
     - `cache_read_input_tokens` — **신뢰 가능**
     - `output_tokens` — 참고
     - ⚠️ **`input_tokens`는 스트리밍 플레이스홀더로 불신 (100x 과소계측)**
4. **ccusage CLI**: `npx ccusage session --json` / `npx ccusage daily --json`

### Step 2: 5축 평가

| 축 | 평가 대상 | 척도 | 핵심 질문 |
|----|----------|------|----------|
| **차크라 총량** | 세션 누적 토큰 (cache+output 합) | E1~E5 | 작업 난이도 대비 토큰이 과다했는가? |
| **순환 효율** | 캐시 히트율 | A/B/C/D | `cache_read / (cache_read + cache_creation)` ≥ 70%인가? |
| **술법 정밀도** | 도구 호출 절제 | ✓/✗ | Read 중복, Grep 반복, 동일 Bash 재실행이 있었는가? |
| **인술 선택** | 모델 적절성 | 1~5 | 간단한 작업에 Opus 쓰지 않았는가? / 복잡 작업에 Haiku 쓰지 않았는가? |
| **분신술 절제** | 서브에이전트 빈도 | Pass/Warn/Fail | 서브에이전트 소환이 작업 복잡도에 부합했는가? |

### 척도 정의

- **차크라 총량 E1~E5**:
  - E5: 최소 (예상 대비 ≤ 1.2배)
  - E4: 양호 (≤ 1.5배)
  - E3: 보통 (≤ 2배)
  - E2: 낭비 (≤ 3배)
  - E1: 차크라 고갈 (> 3배)
- **순환 효율 A~D**:
  - A: ≥ 85% (차크라 순환 최적)
  - B: 70~85% (양호)
  - C: 50~70% (끊김 감지)
  - D: < 50% (순환 파괴)
- **인술 선택 1~5**:
  - 5: 모든 작업에 적정 모델
  - 3: 1회 오선택
  - 1: 반복적 오선택 (낭비 패턴)
- **분신술 Pass/Warn/Fail**:
  - Pass: 서브에이전트 1회 이하 또는 각 소환이 명확한 목적
  - Warn: 2~3회 소환 중 일부가 본인이 처리할 수 있었음
  - Fail: 과다 소환 또는 병렬성 없는 순차 소환

### Step 3: 출력

```markdown
### 🥷 차크라 감사 결과

| 축 | 점수 | 근거 |
|----|------|------|
| 차크라 총량 | {E1~E5} | {총 토큰 / 예상} |
| 순환 효율 | {A~D} | {hit ratio %} |
| 술법 정밀도 | {✓/✗} | {도구 호출 이상치 1줄} |
| 인술 선택 | {1~5} | {모델 사용 내역 1줄} |
| 분신술 절제 | {Pass/Warn/Fail} | {서브에이전트 소환 수} |

**개선 술법 (다음 세션 전략)**:
1. {구체적 권고 1 — 캐시 보존 / 도구 호출 패턴 / 모델 선택 중 하나}
2. {구체적 권고 2}
3. {구체적 권고 3 — 선택적}

**한 줄 요약**: {이번 세션의 차크라 특징 — 예: "분신술 과다로 차크라 낭비"}
```

## 하네스 연동

카카시 하네스(`harness/harness.config.json`)가 존재하면:

- 본 스킬은 `harness/agents/chakra-auditor.md`를 참조하여 동일 규칙을 따른다 (있는 경우).
- 로그는 `harness/logs/chakra-auditor/` 하위에 저장한다.
- 하네스 엔진 실행 후 자동 발동될 때는 하네스의 로그 규약을 따른다.
- `chakra-auditor` 에이전트가 `harness.config.json`의 `agents` 배열에 등록되어 있지 않더라도,
  본 스킬은 단독 규칙으로 감사를 수행한다 (에이전트 정의는 하네스 측 선택사항).

하네스가 없으면:

- 본 스킬 단독으로 작동한다.
- 로그는 `.claude/skills/harness-chakra-kakashi/logs/` 하위에 저장한다.
- `references/` 내부의 규칙 문서만 참조한다.

## 개인정보·보안 보호 (CRITICAL)

JSONL 트랜스크립트에는 사용자 발화 원문·파일 경로·API 키·비밀번호가 포함될 수 있다.
**감사 로그에는 절대 원문을 복사하지 않는다.**

### 기록 금지
- 사용자 메시지 원문, 어시스턴트 응답 원문
- 파일 경로 중 사용자명(`C:\Users\<name>\...`) — 마스킹
- 비밀번호·토큰·API 키 (설령 부분이라도)
- 채팅 앱에서 가져온 대화 내용

### 기록 허용
- 도구 이름과 호출 횟수 (예: `Read x12, Bash x3`)
- 토큰 수치 (cache_read, cache_creation, output)
- 모델 이름 (`claude-sonnet-4-6`, `claude-haiku-4-5` 등)
- 서브에이전트 타입과 횟수
- 소요 시간

자세한 규칙은 `references/chakra-masking.md`.

- 본 스킬은 코드 수정·빌드·배포를 하지 않는다. 감사와 권고만.
